# 测试人员职责清单

## 职责概览

测试人员在测试生命周期中承担以下核心职责：

```
用例设计 → 用例执行 → 结果记录 → 缺陷提报 → 复测验证 → 文档维护
```

---

## 一、测试用例管理

### 1.1 用例设计

| 任务 | 输出 | 要求 |
|------|------|------|
| 分析需求文档 | 用例草稿 | 覆盖所有功能点 |
| 设计测试场景 | YAML 用例文件 | 包含正常/异常/边界场景 |
| 编写断言验证 | assertions 字段 | 明确预期值和验证条件 |
| 确定优先级 | priority 字段 | P0/P1/P2 分级 |
| 添加版本信息 | version/history | 初始版本 1.0 |

**交付物**: `test-cases/*.yaml`

### 1.2 用例维护

| 任务 | 触发时机 | 输出 |
|------|---------|------|
| 新增功能用例 | 新功能上线 | 新模块 YAML + index.yaml 更新 |
| 修改现有用例 | 需求变更/修复后 | version 升级 + history 记录 |
| 废弃无用用例 | 功能取消 | status=deprecated + 说明 |
| 用例评审 | 定期/变更后 | 评审记录 |

**维护原则**:
- ✅ 需求变更 → 修改用例，记录 history
- ❌ 执行失败 → 不修改用例，提交缺陷
- ❌ 实现错误 → 不修改用例，等待修复

---

## 二、测试执行

### 2.1 执行准备

| 任务 | 内容 |
|------|------|
| 确认测试环境 | 操作系统、浏览器、视口配置 |
| 确认测试范围 | 全量/模块/冒烟/回归 |
| 创建执行批次 | `YYYY-MM-DD_run-NNN` 目录 |
| 准备证据目录 | `artifacts/screenshots/videos/logs` |

### 2.2 用例执行

| 任务 | 要求 |
|------|------|
| 按步骤执行 | 遵循 YAML 定义的 steps |
| 记录实际结果 | 每步实际表现 |
| 收集执行证据 | 截图/视频/日志 |
| 判断断言结果 | 通过/失败 |
| 记录异常情况 | 错误信息、复现条件 |

### 2.3 结果记录

| 输出 | 内容 |
|------|------|
| 单用例报告 | `TC-{ID}.md` 每条用例独立报告 |
| 执行汇总报告 | `summary.md` 执行统计、用例列表 |
| 当日索引 | `DATE_index.md` 执行批次汇总 |
| 执行证据 | 截图、视频、日志文件 |

**交付物**: `test-cases/reports/YYYY-MM-DD_run-NNN/`

---

## 三、缺陷管理

### 3.1 缺陷发现

| 任务 | 内容 |
|------|------|
| 确认失败原因 | 是代码缺陷还是用例问题 |
| 分析严重程度 | Critical/High/Medium/Low |
| 定位问题范围 | 代码位置、影响模块 |
| 收集缺陷证据 | 截图、录屏、日志、复现步骤 |

**判断原则**:
```
用例失败 → 分析原因

✅ 确认是缺陷 → 提交 Issue
❌ 确认是用例问题 → 修改用例（需评审）
❌ 确认是环境问题 → 重新执行
```

### 3.2 Issue 提交

| 任务 | 内容 |
|------|------|
| 选择 Issue 模板 | bug_report.md 或 test_failure.md |
| 填写 Issue 内容 | 用例信息、失败详情、证据链接 |
| 设置 Issue 标签 | test-failure, bug, P0, module:xxx |
| 关联测试报告 | 报告文件链接 |

**执行命令**:
```bash
gh issue create \
  --title "[TEST-FAIL] TC-XXX - 失败原因" \
  --body-file reports/DATE_run-NNN/TC-XXX-issue.md \
  --label "test-failure,bug,P0"
```

### 3.3 Issue 追踪

| 任务 | 内容 |
|------|------|
| 监控 Issue 状态 | Open → In Progress → Fixed |
| 确认修复信息 | commit SHA、修复说明、影响范围 |
| 更新报告关联 | 添加 Issue 链接字段 |

---

## 四、复测验证

### 4.1 复测准备

| 任务 | 内容 |
|------|------|
| 监控 Issue 状态 | 关注 Fixed 状态 |
| 获取修复信息 | commit、修复说明、影响范围 |
| 确定复测范围 | 最小/模块/关联/全量 |
| 创建复测批次 | `YYYY-MM-DD_bug-{ID}` 或 `run-NNN` |

### 4.2 复测执行

| 任务 | 内容 |
|------|------|
| 执行相关用例 | 按复测范围执行 |
| 对比首次失败 | 预期值 vs 实际值 vs 首次失败值 |
| 记录对比分析 | 复测报告中的对比表格 |
| 收集复测证据 | 新截图、新日志 |

### 4.3 复测结果处理

#### ✅ 复测通过

| 任务 | 内容 |
|------|------|
| 评论 Issue | 复测通过信息 + 报告链接 |
| 添加标签 | verified |
| 关闭 Issue | `gh issue close {ID}` |
| 更新用例版本 | version 升级（如有变更） |
| 更新当日索引 | 添加复测记录 |
| 通知团队 | 复测通过，可发布 |

#### ❌ 复测失败

| 任务 | 内容 |
|------|------|
| Reopen Issue | `gh issue reopen {ID}` |
| 补充失败信息 | 对比分析、建议排查方向 |
| 添加标签 | regression-failed |
| 通知开发 | 修复无效，需重新处理 |
| 更新当日索引 | 添加失败复测记录 |

## 五、文档维护

### 5.1 报告文档

| 文档 | 维护时机 | 内容 |
|------|---------|------|
| 单用例报告 | 每次执行 | 执行结果、断言验证、证据 |
| 汇总报告 | 每次执行 | 执行统计、用例列表、缺陷 |
| 当日索引 | 每日结束 | 执行批次汇总、缺陷追踪 |
| Issue 状态记录 | 状态变化 | Issue 流转时间线 |

### 5.2 用例文档

| 文档 | 维护时机 | 内容 |
|------|---------|------|
| 模块 YAML | 需求变更/修复后 | version/history 更新 |
| index.yaml | 新增模块 | 模块列表、版本号更新 |
| 流程文档 | 流程优化 | ISSUE/REGRESSION WORKFLOW |

### 5.3 Git 版本管理

| 任务 | 内容 |
|------|------|
| 用例变更提交 | commit + message 说明变更 |
| 报告变更提交 | commit + message 说明内容 |
| 保持历史追溯 | Git 保留所有版本历史 |

| 场景 | 测试动作 |

## 七、日常工作规范

| 规范 | 要求 |
|------|------|

| 执行中 | 按步骤执行、如实记录 |

### 7.2 报告规范

| 规范 | 要求 |

| 关联规范 | Issue 链接、首次失败报告链接 |

## 八、关键原则

### 8.1 用例修改原则

```
✅ 可修改:
- 需求变更导致预期值变化
- 新功能需要新增用例
- 用例设计有明显错误
- 功能取消需废弃用例

❌ 禁止修改:
- 执行失败时修改预期值让其"通过"
- 规避缺陷降低预期标准
- 未确认需求凭个人理解修改
```

### 8.2 Issue 提交原则

```
✅ 必须提交:
- 确认是代码缺陷
- 断言验证失败且有证据

❌ 不应提交:
- 用例设计问题（应修改用例）
- 环境问题（应重新执行）
- 已知缺陷重复提交
```

### 8.3 复测执行原则

```
✅ 复测通过:
- 验证修复有效性
- 对比首次失败确认改善
- 关联范围验证无新问题

❌ 复测失败:
- Reopen Issue（不创建新 Issue）
- 补充对比分析信息
- 建议排查方向
```

## 十、工具使用

### 10.1 Git 操作

```bash
# 查看用例变更历史
git log --oneline test-cases/controls.yaml
git show commit_id:test-cases/controls.yaml

# 提交用例变更
git add test-cases/
git commit -m "更新 TC-XXX 用例 v1.0→v1.1"
```

### 10.2 GitHub CLI

```bash
# 创建 Issue
gh issue create --title "..."

# 查看 Issue
gh issue view 12

# 关闭 Issue
gh issue close 12 --comment "..."

# Reopen Issue
gh issue reopen 12
```

### 10.3 目录管理

```bash
# 创建执行批次
mkdir -p test-cases/reports/2026-04-22_run-001/artifacts

# 查看当日执行
ls test-cases/reports/2026-04-22_*

# 查看特定用例所有执行
find test-cases/reports -name "TC-CTRL-007.md"
```
